# AI実装コーディング規約

本プロジェクトにおいて、AIがソースコードを実装・修正する際に遵守すべきコーディング規約を定義します。本規約は、AIによる品質の安定化に加え、人間による解読とメンテナンスを容易にすることを目的とします。

## 1. 基本方針
- **言語・フレームワーク**: React + TypeScript + Vite を使用。
- **型安全性**: 
    - 型定義（`src/types/`）を適切に行い、可能な限り `any` を排除する。
    - コンポーネントの Props や関数の引数・戻り値には必ず型を明示する。
- **状態管理**: Zustand（`src/store/`）を使用してアプリケーションの状態を管理する。
    - 大規模なストア一つにまとめず、機能ドメイン単位（例：`useTournamentStore`, `useScoringStore`）で分割して定義する。
- **フォーマットと静的解析**: 
    - 既存コードのフォーマット（ESLint / Prettier等の設定）を尊重し、不要なスタイルの破壊や変更を行わないこと。

## 2. メンテナンス性と可読性の向上

### 2.1 命名規則
- **ファイル名**: Reactコンポーネントは `PascalCase.tsx`、フックやロジック・定数ファイル等は `camelCase.ts` を原則とする。
- **コンポーネント**: `PascalCase`（例: `ScoringTable.tsx`）
- **変数・関数**: `camelCase`（例: `calculateTotalScore`）
- **定数**: `SCREAMING_SNAKE_CASE`（例: `DEFAULT_PLAYER_COUNT`）
- **boolean型**: `is`, `has`, `should` などのプレフィックスを付ける（例: `isProcessing`, `hasError`）。
- **カスタムフック**: `use` プレフィックスを付けた `camelCase`（例: `useScoring.ts`, `useLocalStorage.ts`）。
- **ユーティリティ関数**: 機能を表す動詞から始まる `camelCase`（例: `formatDate.ts`, `validateScore.ts`）。
- **ディレクトリ名**: `kebab-case`（例: `feature-scoring`）を基本とする。

### 2.2 自己記述的なコード
- **マジックナンバーの禁止**: 計算ロジック等で特定の固定値を使用する場合、数値を直書きせず `src/constants/` 等に定数（`SCREAMING_SNAKE_CASE`）として定義して意図を明示する。
- **Early Return**: 深いネストを避けるため、エラーチェックや条件不一致の場合は関数の最初で `return` する。
- **ロジックの分離（カスタムフック）**: 
    - 複雑な計算式、データの加工、状態操作等はコンポーネント内に長々と記述せず、カスタムフック（`useScoring.ts` 等）へ切り出す。
    - コンポーネントは「UIの構成とデータの受け渡し」に集中させる。
- **条件分岐の簡素化**: 複雑な条件式は変数に代入して名前を付けるか、専用の関数として切り出す。
- **適切なコメント（JSDoc）**: 
    - 複雑なビジネスロジックや、引数・戻り値の意味が自明でない関数には JSDoc 形式で記述する。
    - コードから意図が読み取れる単純な処理（例：Stateの更新のみ等）には過剰なコメントを付けない。

## 3. コンポーネント分割ルール

### 3.1 ディレクトリ構成
- **`src/components/ui/`**: 
    - ボタン、入力フォーム、モーダル等の、特定のドメイン（採点、選手管理等）に依存しない汎用的なコンポーネントを配置。
- **`src/components/features/[機能名]/`**: 
    - 採点画面、選手設定画面など、特定の機能に関連するコンポーネントを配置。
    - その機能内だけで使われるサブコンポーネントもここに配置する。

### 3.2 分割の基準
- **1ファイル1コンポーネント**: 原則として1つのファイルには1つの主要なコンポーネントのみを記述する。
- **分割のしきい値**: 目安として1つのコンポーネントが **200〜300行** を超える場合や、複数の独立したUIセクション（ヘッダー、サイドバー等）を持つ場合は分割を検討する。

### 3.3 インポートと依存関係
- **機能間の依存回避**: 原則として、`src/components/features/[機能A]/` の内部から `features/[機能B]/` の内部コンポーネントを直接インポートしないこと。複数機能で共有すべきコンポーネントやロジックは `src/components/ui/` や共通レイヤーへ引き上げる。

## 4. デザインの統一性（デザインシステム）

### 4.1 CSS変数（Design Tokens）の徹底
- カラー、角丸、シャドウ、トランジション等のデザイン要素に、ハードコードされた値を使用することを禁止する。
- 必ず `src/index.css` の `:root` に定義されている変数を使用すること（例: `color: var(--color-primary);`, `border-radius: var(--radius-md);`）。
- **z-indexの管理**: `z-index` に直接数値（`999` 等）を記述することを禁止する。必ず `index.css` で定義されたトークン（例: `var(--z-modal)`, `var(--z-overlay)`）を使用すること。

### 4.2 余白（Spacing）の規格化
- 余白（margin, padding, gap）は、原則として **8px (0.5rem)** を基本単位としたスカラーで設計する。
- コンポーネント間の余白には、`index.css` で定義されたユーティリティクラス（`.gap-2` (8px), `.gap-4` (16px) 等）を活用する。

### 4.3 UI部品とクラスの再利用
- 独自のボタン、カード、入力フォームを野良で作らず、`index.css` のグローバルクラス（`.btn`, `.card`, `.form-input`等）をベースとする。
- すでに `src/components/ui/` に存在するコンポーネントがある場合は、それを使用することを最優先する。

### 4.4 インタラクションの統一
- ホバー時やクリック時の色変化、アニメーションの速度などは、`--transition-fast` 等のトークンを用い、システム全体で一貫性を持たせる。
- **アクセシビリティ (a11y)**:
    - 意味を持つアイコン（保存ボタン等）には必ず `aria-label` を付与する。
    - 適切なセマンティックHTML（`main`, `section`, `nav`, `article`等）を使用し、文書構造を正しく保つ。

### 4.5 状態（UI State）のフィードバック
- **ローディング・エラーの統一**: データ読み込み中は共通の `<LoadingSpinner />` やスケルトンUIを使用し、機能ごとに異なるローディング・エラー画面を独自実装しないこと。

## 5. プロジェクト固有の実装ルール

### 5.1 メッセージの集約
- 日本語文字列（ラベル、メッセージ等）は `src/constants/messages.ts` に集約し、ハードコードを避ける。

### 5.2 数値表示の制御
- 数値の並ぶ箇所には必ず等幅フォント（`tabular-nums` 等）を適用する。
- 数値入力時の不要なゼロ埋め（`010` など）は、フロントエンド側で処理して除去する。

### 5.3 破壊的データの操作
- データの初期化や削除を行う際は、必ずバックアップの警告を含む確認ダイアログを挟む。

### 5.4 エラー通知の統一
- バリデーションエラーやAPIエラーが発生した際は、コンソール出力だけでなく、ユーザーに視覚的に伝わるフィードバック（Toast通知、アラート表示等）を行う。
- 通知メッセージは `src/constants/messages.ts` に定義された内容を使用する。

### 5.5 外部通信・データ永続化の分離
- **レイヤーの分離**: LocalStorageへの保存や外部APIとの通信などのデータ永続化処理は、コンポーネントや状態管理（Zustand）の中に直接記述せず、専用のモジュール（例: `src/services/` または `src/api/`）に抽出すること。
